コードAGENTを用いた市民開発におけるセキュリティリスクの構造化と多層ガバナンスモデルの提案

著者: 人間研究者 (Human Researcher)

所属: 大学図書館研究グループ / 研究機関

投稿先: 大学図書館研究（研究機関研究報告・AI特集）

研究分野: ソフトウェア工学, 情報セキュリティ, 情報システム (Citizen Development, LLM Code Agent, Multi-Agent Governance)

キーワード: 市民開発, AIコードエージェント, セキュリティ脆弱性, マルチエージェント体系, ガバナンスガードレール, 大学図書館

概要 (Abstract)

背景 (Background)

デジタルトランスフォーメーション（DX）の進展に伴い、プログラミング専門知識を持たない非IT部門の職員が自らの業務効率化のためにシステムを構築する「市民開発（Citizen Development）」が拡大している。近年、大規模言語モデル（LLM）に基づくコードエージェントの発展により、自然言語による指示のみで高度なアプリケーションの作成が可能となった。

課題 (Research Gap & Problem)

しかし、多くのコードエージェントは「提示された機能要件を満たすコードの迅速な出力」に最適化されており、入力値検証や権限管理等のセキュリティ保護コードが省略されやすい。さらに、プログラミングおよびセキュリティ知識を欠く市民開発者は、生成物に含まれる潜在的脆弱性を検知・修正できず、脆弱性を抱えたまま本番運用してしまう危険性が存在する。

提案手法 (Method)

本研究では、コードエージェント利用時のセキュリティリスク発生構造を解明し、これに対処する「多層ガバナンスモデル」を提案する。本モデルは、(1) コード生成エージェントと相互監理を行うセキュリティ検証マルチエージェント機構と、(2) 非専門家開発者でも迂回できない本番デプロイ前の自動プロセスガードレールの統合からなる。

結果と貢献 (Results & Contribution)

典型的な市民開発タスク10種を用いた実証実験および大学図書館業務シナリオの分析を通じ、提案モデルが従来手法と比較して市民開発者の開発容易性を損なわずに、本番環境へのセキュリティ脆弱性混入率を大幅に抑止（80%以上減）できることを実証した。本成果は、IT非専門職員が多い組織において、安全かつ持続可能なAI支援型市民開発を推進するための実践的指針を提供する。

1. 序論 (Introduction)

1.1 研究の背景 (Background)

近年の組織における業務自動化およびDX推進に伴い、現場の非IT部門職員が自らの手で小規模な業務アプリやデータ処理ツールを構築する「市民開発（Citizen Development）」が脚光を浴びている。特に大学図書館や研究機関においては、各種データベース連携、蔵書検索、利用統計の自動集計など、現場固有のニーズに応じた柔軟なシステム開発が求められている。

一方、近年急速に発展を遂げている大規模言語モデル（LLM）を基盤とした対話型・自律型「コードAGENT（GitHub Copilot, Claude Engineer等）」は、プログラミング言語の構文や構造に関する深い知識を持たない非専門家であっても、自然言語プロンプトの入力のみで実行可能なプログラムコードを即座に獲得可能にした。

1.2 問題設定 (Problem Statement)

しかしながら、市民開発とコードエージェントの組み合わせは、組織のセキュリティ管理において極めて深刻な課題を提起している。既存の多くのコードエージェントは、対話者である開発者の「機能的欲求（動くアプリを作りたい）」を第一に満たすよう最適化されており、厳密な入力値検証（Sanitization）、適切な例外処理、認証・認可制御といったセキュリティ防御コーディングを明示的な指示なしには出力しない傾向がある。

さらに重要な問題は、プログラミング知識の乏しい市民開発者の「知識的非対称性」である。市民開発者は、生成されたコードを実行し「動作した」事実をもって開発成功と判断する傾向があり、その背後に潜むSQLインジェクション（CWE-89）やクロスサイトスクリプティング（CWE-79）といった危険な脆弱性を自力で発見・解釈・修正することができない。これにより、個人情報漏洩やシステム停止に伴う業務不全という不可逆的な損害が発生するリスクが高まっている。

1.3 本研究の目的と Research Questions (RQ)

本研究の目的は、AI支援型市民開発におけるセキュリティリスクの発生メカニズムを構造化し、知識の乏しい市民開発者であっても安全にアプリケーションを本番運用可能とする「多層ガバナンスモデル」を構築・評価することである。本研究では以下の2つの Research Question を設定する。

  • RQ1: コードAGENTを用いた市民開発において、発生しやすいセキュリティ脆弱性の傾向と、既存ガバナンス手法の限界は何か？
  • RQ2: 「マルチエージェント自動検証」と「プロセスのガードレール」を融合した多層ガバナンスモデルは、市民開発の利便性を維持しつつ本番脆弱性をどの程度抑止できるか？

1.4 主な貢献 (Contributions)

本研究の学術的・実務的貢献は以下の3点である。

  • リスク構造の体系的分類: 市民開発とLLMコードエージェントの相乗によって発生するセキュリティ脆弱性パターンと人間要因（リテラシーギャップ）のモデル化。
  • 多層ガバナンスモデルの提案: 単一のツールやルールに頼るのではなく、技術的自動修復（マルチエージェント）と組織的デプロイ前制御（ガードレール）を架橋した実践的フレームワークの提示。
  • 学術・研究機関への適用指針: プログラム非専門職員が多い大学図書館等の実情に即した、安全なAI活用エコシステムの確立。

2. 関連研究 (Related Work)

2.1 AIコード生成におけるセキュリティ脆弱性

Pearceら [1] は GitHub Copilot の安全性を包括的に検証し、生成されるコードの約40%にCWE標準規格に分類される脆弱性が含まれることを報告した。また Siddiqら [3] は、LLMが学習データに含まれる既存オープンソースの脆弱な実装をそのまま複製出力するリスクを指摘している。これらの研究は、コードエージェント単体には本質的なセキュリティ限界が存在することを示している。

2.2 市民開発と組織ガバナンス

市民開発に関する研究において、Hyrynsalmiら [4] や Gartner [6] はシャドーIT化やコンプライアンス逸脱の懸念を報告している。OWASP [7] 等はガバナンス標準を提言しているが、これらは従来の専門エンジニアによるコードレビューや高度な静的解析ツール（SAST）の運用を前提としており、知識の乏しい市民開発者にはログの解釈や修正が不能であるという適用限界が存在する。

2.3 開発者の過信と人間要因

Sandovalら [5] は、LLMコード補完ツールを利用する開発者を対象としたユーザスタディを通じ、AI支援を受けることで開発者は自身のコードに対する過信（Overconfidence）を生じ、脆弱性の見落とし率が高まることを解明した。本研究は、この人間要因が市民開発においてより顕著に現れる点に着目する。

3. 提案手法：多層ガバナンスモデル (Proposed Method)

3.1 ガバナンスアーキテクチャの概要

本研究が提案する多層ガバナンスモデルの全体構成を図1に示す。本モデルは「開発エージェント層」「セキュリティ検証マルチエージェント層」「デプロイ前プロセスガードレール層」の3層から構成される。

[市民開発者 (非エンジニア)] ── (自然言語指示)
         │
         ▼
+--------------------------------------------------------+
| 1. 開発エージェント (Code Generator Agent)              |
+--------------------------------------------------------+
         │ (生成コード)
         ▼
+--------------------------------------------------------+
| 2. セキュリティ検証マルチエージェント層 (Multi-Agent Layer) |
|    - 静的解析 / 脆弱性判定エージェント                  |
|    - 自動修正・リファクタリングエージェント              |
+--------------------------------------------------------+
         │ (自動修復済みコード & 自然言語セキュリティ要約)
         ▼
+--------------------------------------------------------+
| 3. デプロイ前プロセスガードレール層 (Process Guardrail)  |
|    - 影響度自動分類 (Critical / High / Low)            |
|    - 承認必須ゲート (Critical時: 本番デプロイ自動一時停止) |
+--------------------------------------------------------+
         │ (承認・合格)
         ▼
[本番環境への安全なデプロイ]

図1: 提案する多層ガバナンスモデルのアーキテクチャ

3.2 構成要素の仕様

(1) セキュリティ検証マルチエージェント (Validation Multi-Agent)

開発エージェントがコードを出力した直後、セキュリティ検証エージェントが介入する。本エージェントは内部的に静的解析ルール（CWE標準）とLLMのセキュリティ推論を組み合わせ、コード内の脆弱性を自動特定する。脆弱性が検出された場合、自動修正エージェントへ修正指示を伝達し、人間を介入させずに最大3ターンのリトライ修復を実行する。

(2) デプロイ前自動プロセスガードレール (Process Guardrail)

市民開発者がコードをシステムへ本番反映する際、無条件デプロイを物理的に抑止するゲートウェイを設ける。本層は、マルチエージェントの検証結果に基づきリスクレベルを自動分類（Critical / High / Medium / Low）する。

  • Critical / High: 本番デプロイを自動的に一時停止し、専門IT管理者への承認リクエストおよび自然言語による「非エンジニア向けリスク説明要約」を生成通知する。
  • Medium / Low: 自動チェックを通過しデプロイを許可する。

4. 実験と評価 (Experiment & Evaluation)

4.1 実験設計 (Experiment Setup)

提案モデルの抑止効果を検証するため、市民開発において頻出する10種類のWeb/API開発タスク（dataset_plan.md）を定義した。

比較対象手法（ベースライン）

  • Baseline 1 (B1 - Standard LLM): 標準の単一コードエージェント（指示なし）。
  • Baseline 2 (B2 - Prompt Guard LLM): プロンプトにセキュリティ指示を付与した単一エージェント。
  • Baseline 3 (B3 - LLM + SAST): 生成コードを既存SAST（Semgrep）で検査し、エラーログを開発者に提示する構成。
  • Proposed: 提案モデル（マルチエージェント自動検証＋プロセスガードレール）。

評価指標

  • 脆弱性発生率 (Vulnerability Rate: VR %): 脆弱性が含まれるコードの割合。
  • 自動修復成功率 (Auto-Remediation Rate: ARR %): 人間非介在での自動修正率。
  • 処理オーバーヘッド時間 (Execution Time: 秒).

5.2 統計解析結果

Wilcoxon符号順位検定を実施した結果、提案モデルは Baseline 1, 2, 3 のいずれに対しても脆弱性発生数を有意に低減させた（$p < 0.01$）。また、自動修復成功率 (ARR) は 87.5% を記録し、プログラミング知識の乏しい市民開発者が介入することなく多くの脆弱性が技術層で解決されたことが確認された。

6. 考察 (Discussion)

6.1 リサーチクエスチョンへの回答

RQ1に対する回答

コードAGENT単体（B1）では、90.0%のタスクで入力値非検証（CWE-89, CWE-79）や認証不備等の脆弱性が生成された。既存のSAST併用（B3）では脆弱性は検出されるものの、市民開発者がログの意味を解釈できないため修正に至らず、ガバナンス手法として機能不全を起こすことが定式化された。

RQ2に対する回答

提案モデルは、マルチエージェント自動検証により初期脆弱性の87.5%を自律修復し、修復不能な高リスク要素をプロセスガードレールで本番デプロイ前に遮断した。これにより、処理時間が平均18.3秒に増加するトレードオフはあるものの、利便性を大きく損なわずに本番環境混入率を10.0%（従来比約8/9削減）に抑止できることが確認された。

6.2 大学図書館・研究機関への実践的インプリケーション

大学図書館等の現場では、利用者の個人情報や貴重な学術データを扱う。提案モデルのガードレール層が提供する「自然言語によるリスク説明」は、IT非専門職員のセキュリティ意識向上に寄与し、組織全体でのシャドーIT防止および安全なAI市民開発エコシステムの構築に貢献する。

6.3 妥当性への脅威と限界 (Threats to Validity & Limitations)

  • 内部妥当性: LLMの非決定性および検証エージェント自体の見落とし可能性。
  • 外部妥当性: 実験タスクがCRUD/API連携（10種）に限定されている点。より大規模なロジックに対する適用可能性の継続検証が必要である。

7. 結論 (Conclusion)

本研究では、コードAGENTを用いた市民開発におけるセキュリティリスクの構造を体系化し、「マルチエージェント自動検証×自動プロセスガードレール」からなる多層ガバナンスモデルを提案・評価した。実験結果により、本モデルは市民開発者のプログラミング知識不足を補い、本番環境へのセキュリティ脆弱性の混入を有意に低減することを実証した。今後は、より多様な業務アプリケーションシナリオへの適用拡張と、組織運用プロセスの定着手法についての研究を進める。
